## Issues Blocking Proper Mecanum Control

### 1. Wrong IK / Forward Kinematics in use
`velocity_ctrl_mode = true` is hardcoded, so the firmware bypasses IK entirely and takes `u1–u4` directly as per-wheel ω setpoints. The odometry still calls `globalSpeedsFromUOmni` — **wrong for mecanum wheels**, so your pose estimate will drift.

### 2. No cmd_vel → wheel speed conversion in ROS2 node
The ROS2 node has no `cmd_vel` subscriber. You must manually set `u1_desired`–`u4_desired` as ROS2 parameters, with no automatic mecanum decomposition from `vx, vy, omega`.

### 3. Omega calculation is off by 2π
`omegaVals[i] = deltaEncCounts * Rads_per_count / dt / (2π)` — the final `/ (2π)` converts rad/s to **rev/s** (Hz), not rad/s. The PID's desired setpoints (`u_desired`) are then treated as rev/s too, so it's internally consistent, but if you want SI units (rad/s) throughout you should remove that `/ (2π)`.

**Minimal path to mecanum velocity control:**

```c
// Change this:

```
You'll also need `data.x_off` and `data.y_off` to be populated (they're in `InputData` but the sscanf doesn't fill them — set them as constants or add them to the protocol).

**2. Add `cmd_vel` → wheel speed IK to the ROS2 node** — subscribe to `geometry_msgs/Twist`, then for each message compute:
```python
# Mecanum IK (body frame, phi=0 since firmware handles rotation)
lx, ly = 0.15, 0.15  # half wheel base (meters) — tune to your robot
r = 0.05             # wheel radius (meters)
vx = msg.linear.x
vy = msg.linear.y
omega = msg.angular.z

u1 = (vx - vy - (lx + ly) * omega) / r  # front-left
u2 = (vx + vy + (lx + ly) * omega) / r  # front-right
u3 = (vx + vy - (lx + ly) * omega) / r  # rear-left
u4 = (vx - vy + (lx + ly) * omega) / r  # rear-right
# Then set self.message_tx[5:9] = [u1, u2, u3, u4]
```

**3. Tune PID gains** — the default `u_pid` gains in the ROS2 node (`Kp=30.5, Ki=0.5, Kd=1.01`) may need adjustment once the encoder units (rev/s vs rad/s) and motor characteristics are understood.

**Summary:** The firmware and ROS2 node already communicate correctly. The main gaps are: (a) wrong pin in the pinout for encoder 2 (PA15 not PA5), (b) odometry uses omni formulas instead of mecanum, and (c) you need a `cmd_vel` subscriber in the ROS2 node to do automatic mecanum wheel speed decomposition.

=== assistant ===
Yes — for the **I2C bus (BNO055 on PB6/PB7)**. I2C is open-drain, meaning the line is only actively pulled low; it needs external resistors to pull it back high.

**Where:** Two 10kΩ resistors, one from PB6 (SCL) to 3.3V and one from PB7 (SDA) to 3.3V.

The STM32 has internal pull-ups (~40kΩ) that you can enable in software, and looking at your MSP config they may already be enabled, but they're weak. Whether you *need* external ones depends on:

- **Wire length to the BNO055** — longer wire = more capacitance = weaker internal pull-ups struggle to reach logic high fast enough
- **Whether the BNO055 module already has them** — many BNO055 breakout boards (Adafruit, etc.) include onboard 10kΩ pull-ups. If you're using a bare IC you definitely need them; if you're using a module, check its schematic.

If you're seeing I2C communication failures or the `usingIMU` path gives garbage values, missing pull-ups are the first thing to check.

**Other place worth knowing about:** Some H-bridge chips (like the L298N) have floating enable/direction inputs that can cause the motors to twitch at power-on before the STM32 drives them. A 10kΩ pull-down on the PWM enable pins (PA0, PE5, PB14, PF9) to GND ensures the bridges start disabled. Whether you need this depends on which H-bridge chips you're using — the L298N and similar don't have built-in pull-downs.
