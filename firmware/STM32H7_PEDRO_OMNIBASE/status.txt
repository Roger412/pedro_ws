STM32H7_PEDRO_OMNIBASE final status, 2026-05-14

The target firmware was migrated from the original PEDRO L298N/local-PID base toward the OMNIBASE mecanum architecture while keeping the STM32H7_PEDRO source tree untouched. The merged target uses four DC motors through L298N H-bridges, quadrature encoders on TIM1/TIM2/TIM4/TIM8, USART3 at 115200 for the ROS 2 link, and a BNO085 over I2C1 using the SH2 driver stack copied from OMNIBASE. The old MCP2515 and BNO055 paths remain in the tree but are no longer in the active control path.

Architecture: ROS 2 `serial_comm` sends the original 30-float PEDRO command frame at 10 Hz. The firmware parses that frame, runs a 100 Hz FreeRTOS control loop, applies OMNIBASE-compatible mecanum IK/FK with `wheel_sign = {-1,+1,-1,+1}`, drives TIM5/TIM12/TIM14/TIM15 PWM plus PD4/PD5, PD6/PD7, PE2/PE4, PE3/PE6 direction pins, and streams comma-separated `key=value` telemetry back to ROS. Telemetry now includes BNO085 quaternion, gyro, linear acceleration, odometry, encoder speeds, PWM duty, PID gains, and `robot_state`. ROS publishes legacy plotting topics plus `imu/data`, `odom`, `/cmd_vel` input support, and a PEDRO web dashboard.

What works in software: the ROS package builds, the Python nodes syntax-check, the command and telemetry formats cross-check, and the dashboard topic wiring matches the serial bridge. Firmware source review found the intended SH2/BNO085 task, mecanum signs, watchdog, STOP/ESTOP state machine, and debug LED mode in place.

Still TODO: firmware compile validation is blocked because generated CubeIDE Debug makefiles still point at the old `home-custom-base` source paths. ROS lint tests still fail on style/docstring issues. Hardware-dependent items remain: confirm wheel index to FL/FR/RL/RR, verify motor and encoder signs on the lifted robot, confirm BNO085 PD14/PD15/PA4 wiring, measure PEDRO `x_off`, `y_off`, wheel radius, and encoder counts/rev, then bench-test `DEBUG_LEDS=1` before powered drive.
